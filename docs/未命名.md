  关键设计要点：
  1. 配置驱动：API key 等只通过 ir.config_parameter 存（系统设置页统一配），代码里没硬编码
  2. 分类绑定 dataset：每个 diecut.kb.category 自带 dify_dataset_id，同步时自动路由到对应 KB（实现你之前说的"5 个独立知识库"）
  3. 自动迁移：如果文章换了分类，旧 dataset 的文档自动删除，新 dataset 重新创建
  4. 幂等性：内容没变时重复 sync 走 update；遇到 404 not_found 自动转 create（处理 Dify 文档被外部删除的情况）
  5. 观测性：每次同步都写 diecut.kb.sync.log（请求/响应/耗时/错误），失败时 sync_status=failed + sync_error 落盘
  6. 不抛异常：所有失败被 sync 服务吞掉，避免 cron / RPC 调用崩
  7. 状态联动：write() 钩子自动把 synced 改回 pending（除非是归档），保证内容变更被同步识别

  使用流程：
  8. 装/升模块：./odoo-bin -c <conf> -u diecut_knowledge
  9. 在 Odoo 设置页 → 行业知识库 → 填 Dify base_url + Dataset API Key → 测试连接
  10. 在 Dify 控制台为 5 个分类各建一个 Dataset，把 dataset_id 填到 diecut.kb.category 的对应记录
  11. 写一篇测试文章 → 发布 → 点"立即同步到 Dify" → 在 Dify 控制台确认文档出现
  12. 启用 cron 知识库 → Dify：自动同步待处理文章（默认未启用，需要的话改 active=True）
