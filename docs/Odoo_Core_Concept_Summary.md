# Odoo 开发核心知识点解析：依赖与联动机制

## 1. 深度解读：`'seller_ids.price'`
在 Odoo 代码中，这种写法常见于 `@api.depends` 装饰器。

### 核心含义
*   **跨模型监听**：这是一种“点语法”声明。它告诉系统，当前函数不仅依赖于 `seller_ids` 列表的变化（增删），还深入依赖于该列表中**每一行**里的 `price` 字段。
*   **触发机制**：只要任何一个供应商的价格发生变动，Odoo 会自动识别并重新触发关联的计算逻辑。

## 2. Odoo 特有 DSL 与标准 Python 的区别

### 语法本质
*   **字符串而非代码**：在标准 Python 中，`a.b` 是属性访问；但在 Odoo 装饰器中，它是被引号包围的字符串（如 `'a.b'`）。
*   **DSL (领域特定语言)**：这是 Odoo ORM 框架特有的语法。Python 解释器读取它时只是普通字符串，但 Odoo 引擎会将其解析为**数据库依赖网**。

### 核心差异对比
| 特性         | 标准 Python                           | Odoo DSL                                      |
| :----------- | :------------------------------------ | :-------------------------------------------- |
| **思维方式** | **命令式** (手动写如何监听、如何更新) | **声明式** (只需声明依赖谁，系统自动处理更新) |
| **跨表能力** | 需手动建立对象间的信号传递            | 通过“点语法”实现**自动化跨表追踪**            |
| **执行时机** | 即时执行                              | 由 Odoo 后台引擎根据数据库变动信号调度执行    |

## 3. 计算字段 (Computed Fields) 性能解析
计算字段是 Odoo 的灵魂，但使用不当会拖慢系统。

### 性能维度
*   **非存储字段 (`store=False`)**：
    *   **特点**：不占数据库空间，每次读取都重算。
    *   **风险**：在列表视图展示时会产生大量的重复计算。
*   **存储字段 (`store=True`)**：
    *   **特点**：计算结果存入数据库，仅在依赖项变动时重算。
    *   **优势**：大幅提升搜索速度和列表展示性能，是做报表和核算工具的基础。

### 优化黄金法则
1.  **尽量存储**：参与搜索、排序或列表展示的字段务必 `store=True`。
2.  **避免 N+1 查询**：禁止在 `_compute` 循环内执行数据库 `search`，应利用关联对象。
3.  **合理设置 depends**：依赖项设置要精准，避免滥用高频变动字段（如 `write_date`）。

## 4. 模切业务场景应用
将上述技术点应用到我们的原材料标准化核算中：
1.  **标准化引擎**：利用 `product.supplierinfo` 的 `onchange`，将卷价实时拍平为面积价。
2.  **数据底座**：通过 `store=True` 的计算字段，为“独立核算模型”提供稳定的、已折算的 `price_per_m2`。
3.  **动态感知**：利用 `'seller_ids.price'` 的依赖声明，确保采购部每一次调价都能立即反映在财务核算底稿中。
