# Odoo 价格体系与核心逻辑指南

本文档旨在总结 Odoo 中的价格管理逻辑，包括销售、采购、库存及会计核算的底层原理。理解这些逻辑对于正确开发自定义模块（如 `diecut`）至关重要。

## 1. 核心原则（Core Philosophy）

> **"Don't reinvent the wheel. Map to it."**
> **“不要试图创建新的价格字段，而是将你的计算结果映射到 Odoo 的原生字段中。”**

自定义开发的正确姿势是：
1.  在自定义模型中进行特定的业务计算（如：按面积、工艺算价）。
2.  将计算结果 **写入/同步** 到 Odoo 的原生字段（`standard_price`, `list_price`, `supplierinfo`）。
3.  让 Odoo 原生的流程（采购单、库存计价、财务分录）自动读取这些原生字段。

## 2. 价格的三大支柱

### 2.1 销售价格 (Sales Price)

| 字段/模型 | 说明 |
| :--- | :--- |
| **`list_price`** | **主要销售价格**。存储在 `product.template` 上。所有变体共享此基础价格。 |
| **`lst_price`** | **变体特定价格**。这是一个计算字段（Computed Field），逻辑为 `list_price` + `price_extra`（变体额外价格）。 |
| **`product.pricelist`** | **销售价格表**。用于处理复杂的定价策略（如：VIP折扣、数量阶梯价、特定日期促销）。销售订单优先读取价格表，而非直接读取 `list_price`。 |

> **开发建议**：不要为了给特定客户打折而修改 `list_price`，应使用价格表规则。

### 2.2 成本价格 (Cost Price)

| 字段/模型 | 说明 |
| :--- | :--- |
| **`standard_price`** | **标准成本**。存储在 `product.product`（变体）上。这是库存计价和财务核算的**唯一**依据。 |
| **核算方法** | - **标准价 (Standard)**: 手动更新，保持固定。<br>- **平均价 (AVCO)**: 每次采购收货时自动加权平均。<br>- **先进先出 (FIFO)**: 字段显示当前库存的平均成本，但底层记录每批次成本。 |

> **重要**：财务结转成本（COGS）时，只认 `standard_price`。

### 2.3 采购价格 (Purchase Price)

| 字段/模型 | 说明 |
| :--- | :--- |
| **`product.supplierinfo`** | **供应商价格表**。存储在产品表单的“采购”标签页中。 |
| **取值逻辑** | 采购订单 (`purchase.order`) 创建时，会根据 **供应商** + **数量** + **日期**，自动从这里查找匹配的记录填入价格。 |

> **开发建议**：不要在采购单行 (`purchase.order.line`) 上硬编码取值逻辑，而是维护 `product.supplierinfo` 数据。

## 3. 业务流程中的价格流转

### 3.1 采购流程 (Purchasing)
*   **动作**：创建采购订单。
*   **价格来源**：优先查找 `product.supplierinfo`。如果没有，默认为 0（需手动输入）。
*   **注意**：采购单价 **不等于** 成本价 (`standard_price`)。采购价是市场行情，成本价是内部核算价值。

### 3.2 库存收货 (Receiving)
*   **动作**：确认收货 (`stock.picking`).
*   **会计分录**：`借：库存商品`，`贷：收货暂估`。
*   **金额来源**：使用 **采购订单上的价格**。
*   **副作用**：如果是 AVCO 或 FIFO 模式，系统会根据此次收货金额更新 `standard_price`。

### 3.3 销售发货 (Delivery)
*   **动作**：确认发货。
*   **会计分录**：`借：主营业务成本 (COGS)`，`贷：库存商品`。
*   **金额来源**：**严格使用 `standard_price`**（或 FIFO 队列中的成本）。

### 3.4 财务开票 (Invoicing)
*   **供应商账单**：金额来自采购订单（即供应商报价）。
*   **客户发票**：金额来自销售订单（即价格表计算后的售价）。

## 4. 实战：Diecut 模块开发建议

针对模切/卷材业务的特殊性，建议如下数据流：

### 场景 A：原材料（卷材）采购
*   **业务需求**：供应商按“卷”报价，但我们内部可能按“平米”核算。
*   **解决方案**：
    1.  在自定义模块中计算：`单卷价格 = 长度 * 宽度 * 平米单价`。
    2.  将这个 `单卷价格` 写入该产品的 `product.supplierinfo` 记录中。
    3.  **结果**：采购员下订单选这一卷料时，自动带出计算好的总价。

### 场景 B：成品（模切件）成本核算
*   **业务需求**：成品成本 = 材料费 + 刀模费 + 人工 + 机器损耗。
*   **解决方案**：
    1.  在 BOM 或 生产单 (`mrp.production`) 完成时，通过自定义逻辑计算出总成本。
    2.  利用 Odoo 的 **`Landded Cost` (到岸成本)** 或 **生产成本核算逻辑**，将这部分额外费用分摊进产品的 `standard_price`。
    3.  **结果**：根据标准逻辑，发货时的利润分析报表（Margin Report）才会准确。

## 5. 常见误区警示 (Don'ts)

1.  ❌ **不要** 创建 `x_purchase_price`让采购单去读。这会破坏 Odoo 的多供应商管理能力。
2.  ❌ **不要** 试图“实时计算”成本给财务看。财务只看存储在数据库里的静态 `standard_price`。
3.  ❌ **不要** 直接在数据库改 `price_unit` 字段而不触发 `_onchange` 或 `create` 方法，这可能导致关联的税金、小计计算错误。
