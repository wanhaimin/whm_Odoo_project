总体评价                                                                                                                                                      
                                                                                                                                                                
  你们的代码库是我见过的对 LLM Wiki 模式最完整的企业级实现之一。三个模块形成了一个清晰的分层架构：

  原始资料层 (diecut.catalog.source.document + catalog.item)
      → AI 编译层 (KbCompiler + KbEnricher)
          → Wiki 层 (diecut.kb.article + wiki.link + citation)
              → 同步层 (DifyKnowledgeSync + DifyProductSync + QaTicketSync)
                  → 查询层 (AI Advisor + OpenClaw Worker)
      → 治理层 (KbLinter)

  这比大多数 RAG 系统走得更远——你们真的在编译知识，而不仅仅是检索它。

  ---
  ✅ 已经做得好的地方

  1. 三层架构（原始资料 → Wiki → Schema）
  - source.document 作为不可变的原始资料层
  - kb.article 作为编译后的 Wiki 层，有明确的 page type 体系
  - TDS skill JSON 文件和 system prompts 构成了 schema 层

  2. 引用追踪（远超 LLM Wiki 的基本要求）
  - kb.citation 模型记录了 claim_text、source page ref、excerpt、confidence、state
  - 这是真正的 provenance tracking，纯 markdown wiki 做不到这一点

  3. Wiki 图谱（知识间的关系比知识本身更有价值）
  - 带类型的双向边（wiki.link），10 种 link type
  - 编译时自动连接图谱，入链/出链计数
  - 图谱可视化（SVG 力导向图），可以交互式浏览

  4. 治理/检查（Lint）
  - 9 种 lint 检查覆盖了内容长度、引用、孤立页面、过期编译、冲突链接等
  - 支持批量 lint 和 cron 定期检查
  - 每条 lint 问题都有建议的修复方案

  5. 路由计划（资料分发策略）
  - KbCompiler 在编译前先做 routing plan（generate_source_route_plan）
  - 让 LLM 判断资料类型和价值，生成处理方案，人确认后执行
  - 有基于规则的 fallback，不会因为 LLM 失败而完全阻塞

  6. 幂等编译
  - SHA1 content hash，同样的资料不会重复编译
  - SELECT ... FOR UPDATE 行级锁防止并发重复

  7. 品牌综述自动更新
  - 单个产品编译后触发 compile_brand_overview 异步更新品牌综述
  - 这是真正的 compounding knowledge pattern

  8. 多模式 ingest
  - 产品结构化数据 → 产品 Wiki
  - 源文档（PDF/TDS/选型指南）→ 知识 Wiki
  - 多产品对比 → 对比 Wiki
  - AI 对话保存 → 问答 Wiki
  - 品牌汇总 → 品牌综述 Wiki

  ---
  ⚠️ 需要关注的中等问题

  1. System prompts 硬编码在 Python 中（schema 层固化）

  kb_compiler.py 第 16-83 行定义了 5 个 system prompt 字符串。它们包含了 Wiki 文章的结构要求、输出格式约定、知识质量规则等核心 schema 信息。但它们在 Python     
  代码中，修改需要 PR 和升级模块。

  LLM Wiki 的做法是将 schema 放在 CLAUDE.md 中，让 LLM 在运行时读取并遵守。

  影响：调整文章结构（比如要求增加"环保合规"章节）需要改代码。非开发者无法参与 Wiki 规范的迭代。

  2. 没有 index.md / 统一的 Wiki 目录

  在 LLM Wiki 中，index.md 是 LLM 导航 Wiki 的主要入口——每个页面一条记录，有链接和一句话摘要。这里通过 Odoo search 查询 wiki，但没有一个集中的、LLM 可读的目录。

  KbCompiler._build_compile_context() 每次重新搜索相关文章，而不是从一个维护好的索引读取。

  影响：LLM 在编译时花费 token 搜索相关内容。随着 Wiki 增长，搜索的准确性和效率会下降。

  3. Markdown 是二等公民

  kb_article 以 content_html 为主，content_md 通过 _html_to_markdownish() 有损转换得到（kb_compiler.py:971）。这个转换使用了简单的正则替换，会丢失 HTML
  中的表格、结构等信息。而且文章编辑 UI 主要展示 HTML 编辑器。

  LLM Wiki 的核心是纯 markdown 文件，LLM 可以直接读写。

  影响：LLM 阅读文章时读取的是 markdown，但 markdown 质量受限于转换逻辑。如果 LLM 想直接编辑文章内容（比如更新某段描述），它需要理解和修改 HTML。

  4. AI Advisor 保存知识需要用户手动操作

  AI Advisor 的"保存为知识"按钮 ai_advisor.py:save_answer() 会创建 compile_source = "ai_answer" 的文章。但这是一个用户手动触发的操作——LLM Wiki
  建议自动将好的问答沉淀为知识。

  影响：用户可能忘记保存有价值的问答。好答案消失在对话历史中，不会加入 Wiki。

  ---
  🔴 核心差距

  1. Lint 不做矛盾检测

  Lint 检查了 9 种问题，但不检测文章之间的矛盾。虽然 wiki.link 有 contradicts link type，但 lint
  只是统计存在的矛盾链接数量（kb_linter.py:25-31），不会主动发现新的矛盾。

  LLM Wiki 的核心主张：Lint 应该找出"new data contradicts old claims"。当编译新源文档时，如果内容与现有文章存在冲突，系统应该主动标记出来。

  2. 没有查询时 Wiki 合成回答（query-time synthesis）

  所有问答请求都走 Dify。Dify 不知道 Wiki 图谱、不知道文章之间的关联关系、不知道引用链。也就是说：

  - Wiki 的结构化知识（品牌关联、材料类别关系、同系列对比等在 Wiki 中显式维护的关系）在查询时完全不可见
  - 用户在 AI Advisor 中提问时，Dify 返回结果中的引用是 Dify 自己的 chunk 匹配，而不是 Wiki 的 citation 体系

  3. 没有网页搜索集成

  LLM Wiki 描述 lint 时可以："data gaps that could be filled with a web search"。这里完全没有网页搜索能力。

  当编译一篇产品 Wiki 时，如果缺少某个关键参数（比如耐温范围），系统无法主动搜索网页来补充。

  4. Dify 强耦合——Wiki 不能独立运作

  AI 编译（KbCompiler）依赖 Dify Chat API。如果 Dify 不可用：
  - 新产品无法自动生成 Wiki 文章
  - 源文档无法编译
  - 品牌综述无法更新
  - 路由计划只能走规则 fallback

  LLM Wiki 的理想状态是 Wiki 在纯 markdown + git 上就可以运作，LLM API 只是增值层。

  5. CLAUDE.md 是空的，AGENTS.md 只有操作指南

  CLAUDE.md 没有任何内容。AGENTS.md 只有开发环境和构建命令——没有 Wiki 架构描述、没有 page type 约定、没有内容规范。

  这意味着：一个新的 LLM agent 进入项目时，需要从代码中反向工程出所有规则。系统 prompts 中的知识没有在 schema 文档中同步维护。

  ---
  总结

  你们的代码库已经实现了我见过的企业级 LLM Wiki 模式中最完整的管道：从原始资料到 AI 编译、富化、链接、引用追踪、lint 检查、Dify 同步。TDS 处理管线 + KbCompiler 
  编排器 + 双向图谱 这组能力组合在一起，已经解决了一个非常难的问题——让企业知识保持被维护、被关联、可追溯。

  主要的改进空间不在于"加功能"而在于改变重心：从"编译后推到 Dify 去查询"的模式，转向"Wiki 本身就是可查询的知识体"的模式。这意味着：让 LLM 直接读 Wiki
  结构（图谱、链接、引用）来回答问题、让 schema 从代码迁徙到文档、让矛盾检测成为知识演进的核心驱动力。
