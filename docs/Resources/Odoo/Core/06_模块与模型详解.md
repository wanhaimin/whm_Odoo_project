## 1. 模块 vs 模型

### 1.1 简单类比

```
模块 (Module) = 一个完整的 App
模型 (Model)  = App 里的一张数据表
```

**就像:**
```
微信 App (模块)
├── 用户表 (模型)
├── 聊天记录表 (模型)
├── 朋友圈表 (模型)
└── 支付记录表 (模型)
```

### 1.2 核心区别

| 特性 | 模块 (Module) | 模型 (Model) |
|------|--------------|-------------|
| **是什么** | 一个完整的功能包 | 一张数据表的定义 |
| **类比** | 一个 App | App 里的一张表 |
| **文件** | 一个文件夹 | 一个 Python 类 |
| **包含** | 模型、视图、逻辑、权限等 | 字段、方法 |
| **数量** | 一个模块可以包含多个模型 | 一个模型对应一张表 |
| **安装** | 可以安装/卸载 | 随模块一起创建 |
| **依赖** | 可以依赖其他模块 | 可以关联其他模型 |

## 2. 模块详解

### 2.1 什么是模块?

**模块是一个完整的功能包**,包含:
- 数据模型 (models)
- 视图 (views)
- 业务逻辑 (Python 代码)
- 权限规则 (security)
- 数据文件 (data)
- 静态资源 (CSS, JS, 图片)

### 2.2 模块文件结构

```
diecut_custom/              ← 这是一个模块
├── __init__.py
├── __manifest__.py         ← 模块的"身份证"
├── models/                 ← 数据模型文件夹
│   ├── __init__.py
│   ├── diecut_quote.py    ← 包含模型定义
│   └── res_partner.py     ← 包含模型定义
├── views/                  ← 视图文件夹
│   ├── diecut_quote_view.xml
│   └── res_partner_view.xml
├── security/               ← 权限文件夹
│   └── ir.model.access.csv
└── static/                 ← 静态资源
    ├── src/
    │   ├── css/
    │   └── js/
    └── description/
        └── icon.png
```

### 2.3 __manifest__.py (模块的身份证)

```python
{
    'name': 'Diecut Custom',           # 模块名称
    'version': '1.0',                  # 版本
    'category': 'Sales',               # 分类
    'summary': '刀模报价管理',          # 简介
    'description': """
        刀模报价单管理系统
        - 报价单创建
        - 客户管理
        - 报价计算
    """,
    'author': 'Your Company',          # 作者
    'website': 'https://www.example.com',
    'depends': ['base', 'sale'],       # 依赖的其他模块
    'data': [                          # 数据文件
        'security/ir.model.access.csv',
        'views/diecut_quote_view.xml',
        'views/res_partner_view.xml',
    ],
    'demo': [                          # 演示数据
        'demo/demo_data.xml',
    ],
    'installable': True,               # 可以安装
    'application': True,               # 是一个应用
    'auto_install': False,             # 不自动安装
}
```

#### 示例 1: 销售模块 (sale)

```

│   ├── sale.order (销售订单)

    ├── 销售员权限
```

## 3. 模型详解

### 3.1 什么是模型?

**模型是一张数据表的定义**,对应数据库中的一张表。

### 3.2 模型定义示例

```python
# models/diecut_quote.py
from odoo import models, fields, api

class DiecutQuote(models.Model):
    _name = 'diecut.quote'              # 模型名称
    _description = '刀模报价单'          # 描述
    _order = 'date desc'                # 默认排序
    
    # 字段定义 (就像 Excel 表格的列)
    name = fields.Char('报价单号', required=True)
    partner_id = fields.Many2one('res.partner', '客户', required=True)
    date = fields.Date('日期', default=fields.Date.today)
    amount_total = fields.Float('总金额', compute='_compute_amount_total', store=True)
    state = fields.Selection([
        ('draft', '草稿'),
        ('sent', '已发送'),
        ('done', '完成'),
    ], string='状态', default='draft')
    
    # 业务方法
    @api.depends('line_ids.price_subtotal')
    def _compute_amount_total(self):
        for record in self:
            record.amount_total = sum(record.line_ids.mapped('price_subtotal'))
    
    def action_confirm(self):
        self.state = 'done'
```

### 3.3 对应的数据库表

```sql
CREATE TABLE diecut_quote (
    id SERIAL PRIMARY KEY,
    name VARCHAR,
    partner_id INTEGER,
    date DATE,
    amount_total NUMERIC,
    state VARCHAR,
    create_date TIMESTAMP,
    write_date TIMESTAMP,
    create_uid INTEGER,
    write_uid INTEGER
);
```

### 3.4 模型的两种使用方式

#### 3.4.1 创建新模型

```python
# 创建一个全新的模型
class DiecutQuote(models.Model):
    _name = 'diecut.quote'  # 新模型名称
    _description = '刀模报价单'
    
    name = fields.Char('报价单号')
    # ... 其他字段
```

**结果:** 创建一张新的数据库表 `diecut_quote`

#### 3.4.2 继承现有模型

```python
# 扩展现有模型
class ResPartner(models.Model):
    _inherit = 'res.partner'  # 继承现有模型
    
    partner_type = fields.Selection([
        ('customer', '客户'),
        ('supplier', '供应商'),
        ('both', '客户和供应商'),
    ], string='合作伙伴类型')
```

**结果:** 在现有的 `res_partner` 表中添加新字段

## 4. 模块加载顺序

### 4.1 核心规则

**唯一规则:** 按 `depends` 依赖关系

```
被依赖的模块 → 先加载
依赖其他模块 → 后加载
```

### 4.2 简单依赖链

```python
# base 模块 (Odoo 核心)
{
    'name': 'Base',
    'depends': [],  # 没有依赖
}

# sale 模块
{
    'name': 'Sales',
    'depends': ['base'],
}

# my_custom 模块
{
    'name': 'My Custom',
    'depends': ['sale'],
}
```

**加载顺序:**
```
1. base      (最先,没有依赖)
2. sale      (依赖 base)
3. my_custom (依赖 sale)
```

**依赖树:**
```
base
 └── sale
      └── my_custom
```

### 4.3 复杂依赖树

```python
# 模块 A
{
    'depends': ['base'],
}

# 模块 B
{
    'depends': ['base'],
}

# 模块 C
{
    'depends': ['base', 'module_a'],
}

# 模块 D
{
    'depends': ['module_b', 'module_c'],
}
```

**依赖树:**
```
base
├── module_a
│   └── module_c
│       └── module_d
└── module_b
    └── module_d
```

**加载顺序:**
```
1. base
2. module_a (依赖 base)
3. module_b (依赖 base)
4. module_c (依赖 base 和 module_a)
5. module_d (依赖 module_b 和 module_c)
```

### 4.4 循环依赖 (错误)

```python
# 模块 A
{
    'depends': ['module_b'],  # A 依赖 B
}

# 模块 B
{
    'depends': ['module_a'],  # B 依赖 A ❌ 循环!
}
```

**Odoo 会报错:**
```
Error: Circular dependency detected: module_a -> module_b -> module_a
```

## 5. 视图继承顺序

### 5.1 决定因素

视图继承按以下顺序应用:

```
1. 模块依赖关系 (depends) - 最高优先级
2. 视图优先级 (priority) - 其次
3. 视图 ID (数字大小) - 最后
```

### 5.2 Priority 规则

| Priority 值 | 含义 | 应用顺序 |
|-------------|------|----------|
| **数字越小** | 优先级越高 | 越早应用 |
| **数字越大** | 优先级越低 | 越晚应用 |
| **默认值** | 16 | 如果不指定 |

### 5.3 示例

```xml
<!-- 模块 A (priority=10, 先应用) -->
<record id="view_order_form_a" model="ir.ui.view">
    <field name="name">sale.order.form.a</field>
    <field name="model">sale.order</field>
    <field name="inherit_id" ref="sale.view_order_form"/>
    <field name="priority">10</field>
    <field name="arch" type="xml">
        <xpath expr="//field[@name='partner_id']" position="after">
            <field name="field_a"/>
        </xpath>
    </field>
</record>

<!-- 模块 B (priority=20, 后应用) -->
<record id="view_order_form_b" model="ir.ui.view">
    <field name="name">sale.order.form.b</field>
    <field name="model">sale.order</field>
    <field name="inherit_id" ref="sale.view_order_form"/>
    <field name="priority">20</field>
    <field name="arch" type="xml">
        <xpath expr="//field[@name='partner_id']" position="after">
            <field name="field_b"/>
        </xpath>
    </field>
</record>
```

**应用过程:**
```
原始视图:
<field name="partner_id"/>

↓ 应用模块 A (priority=10)

<field name="partner_id"/>
<field name="field_a"/>

↓ 应用模块 B (priority=20)

<field name="partner_id"/>
<field name="field_a"/>
<field name="field_b"/>

最终结果 ✅
```

## 6. 继承机制总结

### 6.1 不冲突的情况 - 所有修改都保留

```xml
<!-- 模块 A: 在 partner_id 后添加 -->
<xpath expr="//field[@name='partner_id']" position="after">
    <field name="field_a"/>
</xpath>

<!-- 模块 B: 在 date_order 后添加 -->
<xpath expr="//field[@name='date_order']" position="after">
    <field name="field_b"/>
</xpath>

<!-- 最终结果: 两个修改都保留 ✅ -->
<field name="partner_id"/>
<field name="field_a"/>        ← 模块 A 添加
<field name="date_order"/>
<field name="field_b"/>        ← 模块 B 添加
```

### 6.2 冲突的情况 - 后者覆盖前者

```xml
<!-- 模块 A (priority=10, 先应用) -->
<xpath expr="//field[@name='partner_id']" position="attributes">
    <attribute name="domain">[('customer_rank', '>', 0)]</attribute>
</xpath>

<!-- 模块 B (priority=20, 后应用) -->
<xpath expr="//field[@name='partner_id']" position="attributes">
    <attribute name="domain">[('is_company', '=', True)]</attribute>
</xpath>

<!-- 最终结果: 模块 B 的 domain 覆盖模块 A ⚠️ -->
<field name="partner_id" domain="[('is_company', '=', True)]"/>
```

### 6.3 什么算冲突?

#### 不冲突的情况 ✅

| 情况 | 示例 | 结果 |
|------|------|------|
| 在不同位置添加字段 | A 在 partner_id 后添加,B 在 date_order 后添加 | 都保留 ✅ |
| 修改不同字段 | A 修改 partner_id,B 修改 date_order | 都保留 ✅ |
| 修改同一字段的不同属性 | A 修改 domain,B 修改 options | 都保留 ✅ |

#### 冲突的情况 ⚠️

| 情况 | 示例 | 结果 |
|------|------|------|
| 修改同一字段的同一属性 | A 和 B 都修改 partner_id 的 domain | 后者覆盖 ⚠️ |
| 替换同一元素 | A 和 B 都 replace partner_id | 后者生效 ⚠️ |

## 7. 实战案例

### 7.1 案例1: 典型的模块结构

```
my_odoo_project/
└── custom_addons/
    └── diecut_custom/              ← 模块
        ├── __init__.py
        ├── __manifest__.py
        ├── models/
        │   ├── __init__.py
        │   ├── diecut_quote.py     ← 模型1: 新建
        │   └── res_partner.py      ← 模型2: 继承
        ├── views/
        │   ├── diecut_quote_view.xml
        │   └── res_partner_view.xml
        └── security/
            └── ir.model.access.csv
```

### 7.2 案例2: 模块依赖关系

```python
# diecut_custom/__manifest__.py
{
    'name': 'Diecut Custom',
    'depends': ['base', 'sale'],  # 依赖 base 和 sale
}
```

**加载顺序:**
```
1. base
2. sale
3. diecut_custom
```

### 7.3 案例3: 视图继承优先级

```xml
<!-- 视图1: 基础修改 (priority=10) -->
<record id="view_order_form_base" model="ir.ui.view">
    <field name="priority">10</field>
    <field name="arch" type="xml">
        <xpath expr="//field[@name='partner_id']" position="attributes">
            <attribute name="domain">[('customer_rank', '>', 0)]</attribute>
        </xpath>
    </field>
</record>

<!-- 视图2: 进一步修改 (priority=20) -->
<record id="view_order_form_advanced" model="ir.ui.view">
    <field name="priority">20</field>
    <field name="arch" type="xml">
        <xpath expr="//field[@name='partner_id']" position="attributes">
            <attribute name="options">{'no_create': True}</attribute>
        </xpath>
    </field>
</record>
```

**最终结果:**
```xml
<field name="partner_id" 
       domain="[('customer_rank', '>', 0)]"  ← 来自视图1
       options="{'no_create': True}"/>       ← 来自视图2
```

```

    ↓

```

## 9. 核心要点总结

### 9.1 模块 (Module)

- 📦 **是什么:** 一个完整的功能包
- 📁 **形式:** 一个文件夹
- 🎯 **目的:** 提供一个完整的业务功能
- 📝 **例子:** sale, stock, diecut_custom
- 🔧 **包含:** 模型 + 视图 + 逻辑 + 权限 + ...
- 🔗 **加载顺序:** 只看 `depends` 依赖关系

### 9.2 模型 (Model)

- 📊 **是什么:** 一张数据表的定义
- 📄 **形式:** 一个 Python 类
- 🎯 **目的:** 存储和管理数据
- 📝 **例子:** sale.order, res.partner, diecut.quote
- 🔧 **包含:** 字段 + 方法
- 🔗 **继承:** 可以继承和扩展其他模型

### 9.3 视图继承

- 🎯 **决定因素:** depends → priority → ID
- ✅ **不冲突:** 所有修改都保留
- ⚠️ **冲突:** 后应用的覆盖先应用的
- 📊 **Priority:** 数字越小越先应用(默认16)

### 9.4 简单记忆

```
模块 = App
模型 = 数据表
一个 App 可以有多张数据表
```
