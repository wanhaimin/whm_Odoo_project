# TDS 检索专用模型训练路线图

## 用户问题
把前面对话里关于“2（TDS 检索专用模型训练路线图）”的最后回复存入 Obsidian。

## 内容
好，那我直接给你一版 **TDS 检索专用模型训练路线图**，按“能落地、能逐步见效”的思路来。

## 先说结论
如果目标是做 **材料/TDS 行业知识库的自然语言检索**，最值得训练的顺序一般是：

**1）Embedding 模型 → 2）Reranker → 3）Query Parser → 4）领域 LLM**

不要反过来。  
因为前 3 个直接决定“能不能搜对”，第 4 个主要决定“会不会说得更像人”。

---

# 一、整体目标拆分
你这个系统本质上要解决 4 件事：

### 1. 用户说人话
例如：
- 找耐高温双面胶
- 有没有 50u 左右的黑色 PET 胶
- 可替代 3M 9448A 的材料有哪些

### 2. 系统能听懂
把这些话转成：
- 行业术语
- 参数范围
- 应用场景
- 替代意图

### 3. 系统能搜准
从海量 TDS、系列页、参数表里把候选找出来。

### 4. 系统能解释
告诉你：
- 为什么命中
- 哪些条件满足
- 哪些条件不确定
- 证据在第几页/哪张表

---

# 二、推荐训练顺序
---

## 第一阶段：先不训练大模型，先把数据底座搭起来
这个阶段甚至可以不用训练，只做工程准备。

## 目标
把原始 TDS/PDF/网页，变成可用于训练和检索的数据资产。

## 你要准备的 6 类数据
### 1. 文档原件
- PDF
- 网页
- 图片/OCR 文本
- 供应商产品页
- 系列选型页

### 2. 文档切片
按层拆分：
- 文档级
- 页面级
- 段落级
- 表格级
- 参数行级

### 3. 结构化字段
抽取：
- 品牌
- 系列
- 型号
- 品类
- 基材
- 胶系
- 厚度
- 颜色
- 剥离力
- 保持力
- 耐温
- 绝缘/导热/阻燃等特性
- 应用场景
- 测试条件
- 单位
- 证据来源

### 4. 术语词典
建立行业同义词库：
- PET = 聚酯薄膜
- 亚克力胶 = acrylic adhesive
- 黑色遮光胶 = black light-shielding tape
- 剥离力 = peel adhesion
- 保持力 = holding power / shear holding

### 5. 单位标准化规则
统一：
- um / μm / mm
- N/25mm / N/in
- ℃ / °F
- min / h / day
- gf / N

### 6. 真实查询日志
收集业务人员真实会怎么问：
- “耐 260 度回流焊的有没有？”
- “总厚度 30u 内、PET 基材、双面”
- “替代德莎某型号”
- “黑色、超薄、模切性好”

这个数据后面是最值钱的训练集来源。

---

# 三、第一类最该训练的模型：Embedding 模型
## 目标
提升“召回能力”，让系统更懂行业语言和应用表达。

## 它解决什么问题
比如用户问：
> 适合电池包固定、耐高温冲击的胶

文档里可能没写“电池包固定”这四个字，但写的是：
- battery bonding
- module mounting
- high heat resistance
- shock resistance

好的 embedding 模型应该能把这种语义连起来。

---

## 训练数据怎么做
训练样本一般做成：

### 正样本
- 查询 ↔ 正确文档片段
- 查询 ↔ 正确型号
- 查询 ↔ 正确参数表行

例如：
- “黑色超薄双面胶” ↔ 某型号介绍页
- “PET 50um 左右双面胶” ↔ 对应参数表
- “摄像头模组固定胶” ↔ 应用说明段落

### 难负样本
- 同类但不满足关键参数
- 同系列但厚度不对
- 都是黑色胶，但不是双面
- 都是 PET 双面胶，但耐温不够

难负样本特别重要，不然模型只会学“看起来差不多”。

---

## 训练目标
让模型更擅长：
- 行业同义词理解
- 应用场景匹配
- 材料类别识别
- 别名/英文/中文混合术语理解

---

## 成果形式
训练后你会得到一个行业 embedding 模型，用于：
- 文档召回
- 段落召回
- 表格行召回
- 相似材料召回

---

## 为什么它排第一
因为检索第一步就是“把候选找出来”。  
如果第一步没召回来，后面 reranker 和 LLM 都没机会发挥。

---

# 四、第二类：Reranker 模型
## 目标
提升排序质量，让“最符合需求的结果”排前面。

## 它解决什么问题
向量检索经常会出现：
- 召回了一堆都挺像的材料
- 但真正最匹配的不在前面

例如查询：
> 黑色、50um 左右、耐高温、适合模切的双面胶

候选可能很多。  
Reranker 的任务是把：
- 参数更接近
- 应用更相符
- 证据更完整
的结果排前。

---

## 训练数据怎么做
样本形式：

- Query + Candidate A → 高相关
- Query + Candidate B → 中相关
- Query + Candidate C → 不相关

或做 pairwise：
- 对同一个 query，A 比 B 更相关

---

## 标注建议
相关性别只标“对/错”，最好分层：
- 3 = 完全符合
- 2 = 基本符合，但有部分参数缺失/偏差
- 1 = 语义相关，但不够精确
- 0 = 不相关

这样 reranker 学得更细。

---

## 它特别适合你的原因
TDS 场景不是“谁沾边谁都行”，而是：
- 要尽量优先展示**最接近筛选条件**的材料
- 避免“看起来很像”的假命中

---

# 五、第三类：Query Parser 模型
## 目标
把自然语言查询转成结构化检索条件。

这是 TDS 检索里非常关键的一步。

例如用户说：
> 找总厚度 50u 左右、黑色、PET 基材、适合摄像头模组固定的双面胶

希望解析成：

```json
{
  "material_type": "double_sided_tape",
  "color": "black",
  "backing": "PET",
  "thickness_um": { "min": 45, "max": 55 },
  "application": ["camera module bonding"],
  "intent": "material_search"
}
```

---

## 它解决什么问题
自然语言里很多表达是模糊的：

- “50u 左右”
- “超薄”
- “耐高温”
- “适合模切”
- “替代 3M 9448A”
- “性价比高一点”

这些不能直接拿去做数据库过滤，必须先解析。

---

## Query Parser 要学的能力
### 1. 实体识别
识别：
- 品类
- 品牌
- 型号
- 应用场景
- 参数名称

### 2. 约束识别
识别：
- 大于/小于/区间
- 左右/接近
- 至少/不超过
- 优先/最好/可接受

### 3. 意图识别
区分：
- 精确筛选
- 替代料查询
- 参数问答
- 材料解释
- 对比分析

### 4. 缺省补全
例如：
- “50u 黑色 PET 胶”
系统可以推测用户大概率问的是双面胶/胶带材料，而不是纯薄膜。

---

## 训练数据怎么做
最好的方式是人工做一批 query → DSL 标注：

```json
{
  "query": "找耐温150度以上的PET双面胶",
  "parsed": {
    "material_type": "double_sided_tape",
    "backing": "PET",
    "heat_resistance_c": { "gte": 150 }
  }
}
```

再慢慢扩充：
- 同义表达
- 模糊表达
- 错别字
- 中英混输
- 口语问法

---

# 六、第四类：领域 LLM
## 目标
让系统更会解释、总结、对比、与人交互。

## 它最适合做什么
- 总结某份 TDS
- 对比两个型号
- 解释某参数的业务含义
- 根据召回结果生成自然语言答案
- 做缺失信息提示
- 生成“建议进一步确认什么”

---

## 它不应该直接负责什么
不要让它独自决定：
- 参数是否满足
- 哪个型号一定可替代
- 数值是否正确
- 单位换算是否可信

这些应该交给结构化层和规则层。

---

## 最合理的位置
领域 LLM 放在系统最后一层，做：
**“解释器 + 答案组织器”**
而不是底层真相来源。

---

# 七、训练数据怎么积累
这是最关键的部分。

## 你至少需要建立 4 套标注集
---

## 1. 检索召回集
格式：
- query
- positive docs/chunks
- hard negatives

作用：
训练 embedding

---

## 2. 排序标注集
格式：
- query
- candidate list
- relevance score

作用：
训练 reranker

---

## 3. 查询解析集
格式：
- query
- structured parse

作用：
训练 query parser

---

## 4. 问答解释集
格式：
- query
- evidence
- answer

作用：
训练领域 LLM 或做 instruction tuning

---

# 八、你怎么知道系统变好了
一定要建立评测集，不然只会靠感觉。

## 建议重点指标
### 检索层
- Recall@10
- Recall@50
- MRR
- NDCG

### Query Parser
- 字段识别准确率
- 范围抽取准确率
- 单位归一准确率
- 意图识别准确率

### 业务层
- Top3 是否出现可用候选
- 假阳性率
- 参数错误率
- 人工复核通过率

对你来说，我认为最重要的是：

- **Top3 命中率**
- **参数误判率**
- **人工复核通过率**

---

# 九、最推荐的落地节奏
## 第 0 阶段：先可用
不训练，只做：
- OCR/解析
- 参数抽取规则
- BM25 + 向量检索
- 证据回链

目标：先做出能搜的版本。

---

## 第 1 阶段：训 embedding
目标：解决“召回不全”。

你很快就会感受到提升。

---

## 第 2 阶段：训 reranker
目标：解决“搜出来了但排序不对”。

这个阶段用户体验会明显好很多。

---

## 第 3 阶段：训 query parser
目标：把自然语言查询变成真正可控的过滤条件。

到这里系统就开始“专业”了。

---

## 第 4 阶段：训领域 LLM
目标：把最终回答做得更自然、更懂业务。

这一步是锦上添花，不是雪中送炭。

---

# 十、如果预算和资源有限，最值的组合是什么
我给你一个非常现实的建议：

## 最优先投入
- 数据清洗
- 参数标准化
- 术语词典
- 真实查询标注
- embedding 微调
- reranker 微调

## 次优先投入
- query parser
- 规则引擎
- 替代料逻辑

## 最后再考虑
- 超大领域 LLM
- 本地 200B 级部署

---

# 十一、你这个行业特别值得做的“专用能力”
除了通用检索，我觉得你还应该专门做这几类：

## 1. 替代料检索
用户问：
- 可替代 3M 9448A 的材料有哪些？

这不是普通语义检索，而是：
- 找功能接近
- 参数接近
- 应用接近
- 风险可说明

这是非常高价值能力。

---

## 2. 参数缺失容忍检索
很多 TDS 不完整。  
系统要能区分：

- 明确满足
- 明确不满足
- 信息不足待确认

别把“没写”当“满足”。

---

## 3. 测试条件敏感检索
比如剥离力、保持力一定要带测试条件。  
系统最好能识别：
- 数值
- 条件
- 标准方法
- 是否可横向比较

---

## 4. 系列-型号联动检索
有些 TDS 是系列页，不是单型号页。  
系统要知道：
- 系列级信息
- 型号级差异
- 哪些字段可继承，哪些不行

---

# 十二、最后给你一个一句话路线图
你可以把这件事理解成：

> 先把 TDS 变成“可检索、可过滤、可验证”的数据，再训练 embedding 提升召回、训练 reranker 提升排序、训练 query parser 理解自然语言，最后再用领域 LLM负责解释和交互。
