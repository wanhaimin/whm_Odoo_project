# Odoo 深度方案：基于“批次属性”的模切库存管理 (Lot-Based Dimension Management)

这个方案的核心哲学是：**产品 (Product) 只是一个类别，批次 (Lot/Serial) 才是实物。**

在模切行业，`3M 9448` 本质上是一种材料，它的**材质**是不变的，变的是它的**物理形态 (长 x 宽)**。因此，我们将“长宽”定义为“物理形态属性”，挂载在批次上。

---

## 1. 数据模型改造 (Model Design)

我们需要在 Odoo 的原生 **批次/序列号模型 (`stock.lot`)** 上进行扩展。

### 1.1 修改 `stock.lot` 模型
我们需要给每一个批次号添加两个关键字段：
*   `x_width`: 宽度 (mm)
*   `x_length`: 长度 (m)
*   `x_area`: 面积 (㎡) —— *这是一个计算字段: `width/1000 * length`*

> **为什么加在这里？**
> 因为库存里每一卷特定的胶带，都有一个唯一的批次号（或者条码）。扫一下条码，系统就应该知道这一卷是 5mm 宽还是 10mm 宽。

### 1.2 修改 `product.product` 模型
产品本身的主单位建议设为 **平方米 (m²)** 或者 **千克 (kg)**，因为这是财务核算成本的基准。
*   同时，产品上可以增加一个布尔值 `x_is_roll_material` (是否卷材)，用来控制界面逻辑。

## 2. 业务流程演示 (Workflow)

### 场景一：原材料采购入库 (母卷)
1.  **采购订单**: 购买 `3M 9448`，数量 `100 ㎡`。
2.  **收货入库**:
    *   仓库员收到 2 卷，每卷宽 1000mm，长 50m。
    *   **录入操作**:
        *   创建批次 `LOT_20231201_A`: 输入宽度 `1000`, 长度 `50`。
        *   创建批次 `LOT_20231201_B`: 输入宽度 `1000`, 长度 `50`。
    *   库存增加: `100 ㎡` (分布在两个批次上)。

### 场景二：分切与模切 (生产)
客户下了一个单子，要 100 卷 `5mm 宽` 的胶带。

1.  **生产工单 (Manufacturing Order)**:
    *   **投入组件**: `3M 9448` (母卷)。
    *   **产出成品**: `3M 9448` (子卷)。
    *   *注意：这里投入和产出是同一个产品 ID，只是形态变了。*

2.  **领料 (Consumption)**:
    *   工人扫描母卷条码 `LOT_20231201_A`。
    *   系统识别：这是一卷 1000mm 宽的母卷。
    *   **分切逻辑**: 工人把这卷母切成了 100 个 5mm 宽的小卷，以及剩下了 500mm 的余料。

3.  **完工产出 (Output)**:
    *   系统自动生成（或批量生成）100 个新批次号：`SLIT_001` 到 `SLIT_100`。
    *   这些新批次的属性自动写入：`x_width = 5mm`, `x_length = 50m`。
    *   同时生成一个余料批次 `REST_001`：`x_width = 500mm`。

### 场景三：库存查询
老板问：“仓库里还有多少 5mm 宽的 9448？”
*   打开库存列表 -> 搜索产品 `3M 9448`。
*   增加二次筛选：`宽度 = 5mm`。
*   系统列出所有符合条件的批次，并合计总数量。

## 3. 技术实现难点与对策

### 3.1 单位换算 (UoM Conversion)
*   **难点**: 仓库员习惯数“卷”数，但系统账上记的是“平方米”。
*   **对策**: 在界面上增加一个辅助输入框。
    *   *输入*: `10` 卷, 规格 `5mm * 50m`。
    *   *系统自动计算*: `10 * 0.005 * 50 = 2.5 ㎡`，并把这个数字记入库存账。

### 3.2 标签打印
*   Odoo 的标签打印非常灵活。
*   我们可以定制 ZPL 模板，二维码里包含：`产品ID` + `批次号`。
*   扫码枪一扫，既知道是什么，也知道多宽多长。

---

## 4. 总结
**“批次属性方案” = 给每一卷材料发一张独一无二的身份证。**

*   **极度灵活**: 再也不怕客户的奇葩尺寸要求。
*   **余料管理**: 哪怕只剩 2 米长的尾料，系统也记录得清清楚楚，下次小单子优先用余料，**极大降低损耗成本**。
*   **门槛**: 需要一定的开发量（给 stock.lot 加字段，写单位换算逻辑），但这正是 Odoo 二次开发的价值所在。
