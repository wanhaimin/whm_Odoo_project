# Phase 4 切换准备清单

本文档用于在开发环境完成 Phase 1/2/3 后，推进到可控切换（Cutover Ready）状态。

## 1. 目标

- 在不影响现网业务连续性的前提下，完成从旧目录模型到新目录模型的切换准备。
- 保持随时回滚能力（统一入口可切回 `legacy_split`）。

## 2. 变更边界

- 本阶段不删除旧模型及旧入口。
- 本阶段不做不可逆数据迁移。
- 仅做字段对齐、行为对齐、权限对齐和验收脚本化。

## 3. 字段映射清单（必须确认）

对每个字段给出以下信息：

- 业务含义
- 旧模型字段（`product.template` / `product.product`）
- 新模型字段（`diecut.catalog.item`）
- 当前主数据源（旧/新）
- 同步方向（单向/双向）
- 冲突策略（以谁为准）

建议先覆盖以下关键字段：

- 目录标识与层级：`item_level`、`parent_id`、`is_system_default_series`
- 品牌与分类：`brand_id`、`categ_id`
- 编码与名称：`series_code`、`code`、`name`
- 状态与发布：`catalog_status`
- ERP 映射：`erp_enabled`、`erp_product_tmpl_id`
- 追溯关系：`legacy_tmpl_id`、`legacy_variant_id`

## 4. 数据一致性门槛（Go/No-Go）

切换前必须全部满足：

- `missing_shadow_count = 0`
- `duplicate_brand_code_count = 0`
- `orphan_model_count = 0`
- 新旧关键字段抽样一致率 >= 99%（可配置）
- 关键业务路径人工验收通过（新增、编辑、检索、启用）

建议保留连续 3 次对账结果（间隔至少 1 天）均通过。

## 5. 读写归属策略

建议执行顺序：

1. 当前阶段：读旧（默认），新灰度可读；写通过服务层双写。
2. 预切换阶段：统一入口切到新灰度给管理员试运行，普通用户仍读旧。
3. 切换阶段：统一入口切新，保留旧入口只读可回查。
4. 稳定阶段：评估下线旧入口和旧同步逻辑（单独变更单）。

## 6. 权限与可见性

- 普通用户：仅可见旧架构入口。
- 管理员：可见实验入口、健康检查、统一入口切换。
- 切换窗口期间，统一入口切换权限只保留给指定负责人。

## 7. 回滚 SOP（必须演练）

### 7.1 触发条件

- 切换后出现核心流程失败。
- 新旧数据差异超过阈值。
- 权限异常导致关键人员不可用。

### 7.2 回滚步骤

1. 统一入口切回 `legacy_split`。
2. 管理员确认旧入口可用（列表、表单、保存）。
3. 运行一次影子对账并记录结果。
4. 暂停新架构写入（如需，临时关闭双写入口）。

### 7.3 回滚验收

- 旧入口可完整执行 3 条关键业务路径。
- 近 24 小时新增/修改数据可追溯。

## 8. 冒烟脚本（建议）

每次部署后执行：

- 模块升级：`odoo -u diecut`
- 统一入口模式检查：应为 `legacy_split`
- 影子对账：缺失/重复/孤儿应为 0
- 菜单权限：普通用户不可见实验入口

## 9. 交付物

- 字段映射表（可用 Excel 或 markdown 表）
- 切换窗口执行单（时间、负责人、验证结果）
- 回滚演练记录
